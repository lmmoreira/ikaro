# M13 — Dashboard Frontend: Implementation Details (Developer Reference)

> **Who this is for:** A developer continuing work on `apps/web`'s dashboard/customer-account/hotsite areas who wants to understand *why* M13 is built the way it is — not just what files exist. Pairs with `docs/lean/WEB_LEAN.md` (concept-by-concept learning journal) and `plan/M13-DASHBOARD-FRONTEND_IMPLEMENTATION_DETAILS_IA.md` (token-efficient artifact reference). M13 is the largest milestone in this codebase to date — 44 stories, spanning the entire staff/manager dashboard, the customer account area, the hotsite editor, and the Playwright E2E infrastructure everything after it relies on.

## The Transport Rewrite: Why S01's Design Didn't Survive to S20

S01 shipped a conventional-looking API client: a module-level `configureBffClient({ token, tenantSlug, tenantId })` singleton, an Axios instance that read from it to inject `Authorization: Bearer <token>` and `X-Tenant-Slug` headers on every request, and a `QueryProvider` wrapping the app.

**The bug that killed it:** `tenantId` was read during React's render phase, but the singleton was populated inside a `useEffect` — which runs *after* the first render. Every component's first paint saw `tenantId === ''`. This is a classic "client-side singleton fighting React's render/effect ordering" problem — the kind of bug that's invisible in a quick manual test (by the time you look at the screen, the effect has already run) but shows up as a flash of wrong data, a failed first request, or a hydration mismatch depending on exactly what raced.

**The fix, landed quietly inside S17's PR ("centralize BFF transport"):** move tenant identity out of client-side JS entirely.

- The BFF issues an **httpOnly cookie** (`access_token`) instead of returning a JWT in the response body for the client to store and re-attach.
- `bff-client.ts` sends it automatically via `withCredentials: true` — no manual header injection, no singleton, nothing to race.
- `TenantProvider` (`providers/tenant-provider.tsx`) is populated **server-side**: each dashboard/customer layout decodes the JWT (already sitting in the request's cookies) and passes `tenantId`/`tenantSlug` down as props before the client ever renders. `useTenant()` just reads context — no fetch, no race, correct on the very first paint.
- A new `bffServerFetch(token, path)` helper covers server-side calls (Server Components, Route Handlers) that need to forward the cookie's JWT explicitly, since `fetch()` in a Server Component doesn't automatically carry the browser's cookies the way `axios`'s `withCredentials` does client-side.

**Backend analogy:** this is the same lesson as "don't put request-scoped state in a singleton" that motivated `RequestContext`'s `AsyncLocalStorage` design on the NestJS side. `configureBffClient()` was a singleton trying to hold *per-request* (really, per-render) state — the browser-side equivalent of a service accidentally sharing mutable state across concurrent requests. The fix in both cases is the same shape: stop mutating a shared container and instead thread the value through the call itself (server-side props / `AsyncLocalStorage`), or hand it to something that's naturally request-scoped (an httpOnly cookie the browser manages for you).

One side effect worth knowing: `TenantGuard` on the BFF (`shared/guards/tenant.guard.ts`) still checks `X-Tenant-Slug` *if present*, but since the browser stopped sending it after S17, the guard is now effectively a no-op for every real request. It wasn't deleted — tenant scoping moved entirely into the JWT/cookie, and the header check became dead-but-harmless code. If you're auditing this guard, don't be confused into thinking it's actively enforcing anything today.

## The Mobile Nav Trap: An Invariant That Broke by Copy-Paste

`BottomNav` (the staff/manager dashboard's mobile tab bar) hides itself on certain routes — this is intentional: booking detail, service detail, loyalty detail, and team detail/invite are all **drill-down** pages (you navigated *into* a specific record from a list), and each has its own topbar back-arrow as the way out. Hiding the global nav there is fine because the user always has an escape route.

S31 (Settings) and S35 (Hotsite editor) each added themselves to this same hide-list — following the established pattern without re-deriving *why* the pattern existed. That was the bug: Settings and Hotsite are **not** drill-downs. They're top-level sections, reached directly from the sidebar/bottom-nav "Mais" sheet, exactly like Agenda or Serviços. They have no topbar back-arrow, because there's nothing to "go back" to — you're already at the top level.

The consequence: on a narrow viewport, a manager opening `/dashboard/settings` or `/dashboard/hotsite` had **no way to reach any other section of the app**. The bottom nav was hidden (following the drill-down pattern), and there was no back arrow (because these aren't drill-downs) — the only way out was a manual URL edit or the browser's back button. This went unnoticed through code review and CI because nothing about it is a *type error* or a *failing test* — the page renders fine, the settings form works fine, the bug only exists as an interaction: try to tap "Agenda" from the settings screen on a phone.

**The fix has two parts, and both matter:**

1. Remove Settings and Hotsite from `BottomNav`'s hide-list — they render the nav again.
2. But each page *also* has its own fixed-position mobile action bar (Settings' Save button, Hotsite's Publish/Preview buttons) sitting at `bottom-0` — the exact same screen position `BottomNav` now occupies again. So both pages' action bars were moved to a shared offset, `MOBILE_ACTION_BAR_CLEARANCE_CLASS` (`bottom-[calc(4rem+env(safe-area-inset-bottom,0px))]`), stacking them *above* `BottomNav` instead of overlapping it.

**The generalizable lesson:** a hide-list (or any allow/deny list) encodes an *implicit rule* about the routes in it — here, "this route has another way out." When you add a new entry by pattern-matching the existing ones, you're not just adding a route, you're implicitly asserting that the same rule holds for it. That assertion is exactly the thing that's easy to skip when the list already has six entries that all *do* satisfy the rule and you're the seventh person adding to it. The fix worth remembering isn't "don't hide nav on top-level pages" (too specific) — it's "before adding to an existing list, ask what invariant every current member satisfies, and confirm your new entry actually satisfies it too, not just that it superficially resembles the others."

## "Derived, Not Stored": One Principle, Three Bugs, Same Root Cause

Three separate values in M13 — a booking's cancellation deadline, the loyalty points-to-currency conversion rate, and a staff member's activation status — all hit the *same* class of bug, independently, in different stories. Once you see the pattern, it explains a handful of "why is this computed way over there instead of just being a column" decisions throughout the codebase.

**Attempt #1 (all three): compute it in the BFF.** The BFF is a natural place to reach for — it already aggregates data for the frontend, and it's "closer" to the client. For `cancellableUntil`, the BFF tried computing `scheduledAt − cancellationWindowHours` itself, which meant fetching the tenant's `cancellationWindowHours` setting via `GET /tenants/settings`. For `conversionRate`, same story — fetch the setting, do the arithmetic in the mapper. For staff status, the analogous mistake was checking `isActive` before `googleOAuthId` in the derivation function — not a BFF-vs-backend issue, but the same shape of "the derivation logic silently assumed something about ordering/availability that wasn't guaranteed."

**Why the settings-fetch approach broke:** `GET /tenants/settings` is guarded `STAFF`/`MANAGER`-only. It has to be — it's the same endpoint the Settings *form* reads, and a customer has no business seeing a tenant's internal configuration. But `cancellableUntil` and `conversionRate` both need to be computed for **customer-facing** responses (a customer viewing their own booking, a customer checking their own loyalty balance). The BFF, acting on behalf of a customer, called a staff-only backend endpoint — and got a `403`. This wasn't caught in the stories that introduced the computation, because no customer-facing UI existed yet to exercise the code path; it surfaced later, when S27 (customer account) and S26 (completion loyalty strip) actually wired up the calls.

**The fix, both times: move the computation to the backend, at the source, using data already in scope.** The backend's `GetBookingByIdUseCase` already has `RequestContext.settings` available (it's populated for every authenticated request, not gated by role) — so `Booking.cancellableUntil()` lives on the aggregate itself, computed once, returned identically regardless of whether the caller is staff or customer. Same for `conversionRate`: the backend's loyalty controller reads `RequestContext.settings.loyalty.pointsPerCurrencyUnit` directly and attaches it to the balance response. The BFF's role shrank back to what it should have been — pass the number through, don't compute it.

**Backend analogy, if you're coming from a more traditional layered-architecture background:** this is the "don't put business logic in the presentation/API-gateway layer" rule, just with an extra twist — the naive fix ("call the settings endpoint from wherever you need the setting") looks reasonable until you notice that *not every caller is allowed to call that endpoint*. The real fix isn't "give the BFF broader permissions" (that would defeat the point of the guard) — it's "the thing that already has unconditional access to the setting should do the computation and hand back a result, not the setting itself."

**The staff-status bug is the same shape without an HTTP guard involved.** `deriveStaffStatus(googleOAuthId, isActive)` needs both fields to classify a row correctly: `googleOAuthId === null` means "never completed OAuth" (`PENDING`), otherwise `isActive` decides `ACTIVE` vs `DEACTIVATED`. S13 changed `Staff.invite()` to provision `isActive=true` immediately (closing a real security bug where a not-yet-activated invite could be bypassed) — but the status-derivation function elsewhere in the codebase still checked `isActive` *first*. Since every fresh invite now has `isActive=true`, every fresh invite was misclassified as `ACTIVE` instead of `PENDING`. The fix was purely a check-ordering swap, but the *lesson* is the same as the other two: when a field that used to be a reliable signal (`isActive=false` meaning "not yet activated") changes what it means as a side effect of an unrelated fix, every downstream place that read the old meaning needs to be re-audited — grep for the field, don't assume the one call site you're editing is the only one.

## The Completion Transaction: Why Two Use Cases Became One

Loyalty points are earned when a booking completes, and a customer completing that same booking might also *redeem* points for a discount in the same action ("mark complete, and apply a 200-point discount"). The natural way to build this, given two existing capabilities (`RecordLoyaltyEntriesUseCase` for earning, `RedeemPointsUseCase` for spending), is to call both from a new orchestrating use case, one after the other.

That's what the first draft of S11 did, and it has a real atomicity hole: if the process crashes *between* the redemption write and marking the completion event as processed, retrying the whole flow on redelivery (Pub/Sub is at-least-once — this is a documented invariant of the event bus, not a hypothetical) redeems the points a second time. Two separate use cases mean two separate transactions (or an awkward shared-transaction composition that fights each use case's own transaction-management), and two separate idempotency checks that don't compose into one atomic guarantee.

**The fix:** `CompleteBookingLoyaltyEffectsUseCase` is a single, self-contained use case that inlines both the earn and the optional redeem logic under **one** `txManager.run()` block, with **one** idempotency check keyed by `(eventId, CONSUMER_NAME)`. `RedeemPointsUseCase` wasn't deleted — it still serves the standalone admin `POST /loyalty/redeem` endpoint, where "redeem points, independent of any booking completing" is a real, separate operation with its own correct atomicity boundary. The lesson isn't "never compose use cases" — it's "if two operations must succeed or fail *together* as one unit, they need one transaction and one idempotency key, not two coordinated ones." Composing existing use cases is fine when their outcomes are genuinely independent; it's the wrong tool when the whole point is that they can't be allowed to diverge.

## E2E Test Infrastructure: The Foundation Every Later Story Builds On

S41 is a small-sounding story ("Playwright E2E infrastructure") that ended up being the most structurally important story in the milestone for anything that came after it — including the hotsite editor's E2E suite, added in the very last story (S37).

**Auth in tests is dev-login, never real Google OAuth.** Testing against real OAuth would mean either mocking Google's servers or maintaining real test Google accounts — both fragile and slow. Instead, the BFF has a `POST /auth/dev-login` endpoint, gated behind `ENABLE_DEV_AUTH=true` and an internal API key, that mints the exact same `access_token` cookie a real login would produce — same JWT shape, same claims, same cookie options. `apps/web/e2e/helpers/auth/{staff,customer}-login.ts` wrap this into `loginAsStaff(page, email, tenantSlug)`/`loginAsCustomer(...)`. From the app's perspective, a dev-logged-in session is indistinguishable from a real one; only the *how you got the cookie* differs.

**A subtle trap this surfaced:** Playwright's `page.request` — the API used to make raw HTTP calls from within a test, e.g. to seed fixture data before the test's actual UI interactions — shares the *page's* cookie jar. If a helper needs to log in as a *different* user purely to set up state (for example, linking a staff member's Google account as part of seeding a "this staff member is already active" fixture, without wanting to actually be logged in as them for the test), and that helper calls dev-login through `page.request`, the dev-login response's `Set-Cookie` silently overwrites the *test's own* session cookie. The test then continues, believing it's still logged in as the original user, but every subsequent request is actually authenticated as the fixture user — manifesting as confusing, intermittent 403s that have nothing obviously to do with cookies. The fix is mechanical once you know the cause: any fixture-seeding call that logs in as someone other than the test's primary user must use an isolated `playwrightRequest.newContext()`, never the shared `page.request`.

**The `data-testid`-only convention wasn't a day-one decision — it was learned the hard way.** The original S41 PR needed four follow-up commits (each titled something like "remove translated strings from locators") before the convention actually stuck across the codebase. The failure mode it prevents: a test written with `page.getByText('Salvar')` passes today, then silently breaks the moment someone adds the `pt-BR`/`en` locale split and the button text becomes `t('save')` — or breaks for the *opposite* reason, when a copy-editing pass changes "Salvar" to "Salvar alterações" for UX reasons totally unrelated to the test's actual concern (does clicking the save button trigger the right mutation?). `data-testid` decouples "which element is this" from "what does it currently say," which is exactly the separation a test should have. Three lint rules now enforce this mechanically (`scripts/pre-pr.sh`'s E2E-1/E2E-2/E2E-3): no `getByLabel`/`getByText`, no date literal baked into a `data-testid` value, no template-literal `data-testid` at all — every variable part gets its own `data-*` attribute instead of being string-interpolated into the id.

**Why `playwright.config.ts` has no `webServer` block:** many Playwright setups have the config itself boot the app under test. This repo's tests instead assume the *whole* stack (backend, BFF, web, Postgres, Pub/Sub emulator, GCS emulator) is already running — locally via `docker compose up -d && pnpm dev`, in CI via a dedicated `pr-e2e.yml` workflow that does the equivalent. This is a deliberate trade-off: a monorepo with three independently-running services doesn't fit neatly into "one config starts one server," and trying to make Playwright orchestrate all of it would just be reimplementing what `docker-compose` and the CI workflow already do well. The cost is that you can't `npx playwright test` cold — you need the stack up first — but that's true of this whole codebase's development model already (CLAUDE.md's local-verification gate exists partly because "start the stack" is a real, non-trivial action here, not a one-line `next dev`).

## The Hotsite Editor: SEO Value Objects and the Publish/Preview/Image Triangle

S35-S37 finished the hotsite editor that M12 had only laid the public-facing groundwork for. The most instructive part is how three seemingly separate features — SEO validation, Preview, and image handling — turned out to be tightly coupled once actually built.

**SEO limits became value objects, not just tighter numbers.** The original spec used round placeholder numbers (70/160 characters). During implementation, these were corrected to the numbers that actually matter — 60 and 158 characters, the points at which Google truncates a search result's title and description in practice. Rather than just changing two `.max()` calls, the fix promoted `title`/`description` to dedicated `SeoTitle`/`SeoDescription` value objects (mirroring the existing `HexColor` VO's `create()`/`reconstitute()`/`value` shape) — because a bare length check scattered across the DTO, the aggregate, and the BFF schema is exactly the kind of validation that drifts silently when one copy gets updated and the others don't (this repo has hit that class of bug before, hence the VO convention in the first place).

That promotion surfaced a second, easy-to-miss bug: `HotsiteConfig.updateContent()` re-validated `seo` on *every* call, including calls where `seo` wasn't being changed at all — the use case always passes the current stored value through if the caller didn't supply a new one. If any tenant's stored SEO title happened to be, say, 65 characters (perfectly valid under the *old* 70-char limit, saved before this story ever shipped a UI for editing SEO — but the backend endpoint for it existed already), tightening the limit to 60 would make `updateContent()` throw on **every subsequent update to that tenant's hotsite, even a branding-only change that never touches SEO at all**. The fix: only re-validate when the incoming value actually differs from what's already stored. This is a narrower instance of a broader rule worth generalizing — tightening a validation rule on data that's already persisted is not "just" a frontend/DTO change; it's implicitly a data migration question, and "don't re-validate what you're not changing" is often the cheapest way to avoid needing an actual migration.

**Preview and Publish turned out to need the same image-URL fix, in opposite directions.** The backend resolves every stored image field (branding logo, module background images, gallery/testimonial photos) to a full public URL on `GET` — convenient for rendering, but `PATCH` requires the *raw* storage path (`tenants/<id>/hotsite/...`), because that's what the backend's own upload-verification logic checks against. Two directions, one underlying fact (the raw path is always recoverable from a resolved URL by finding the `tenants/<id>/` marker inside it, since the resolution is a deterministic string concatenation, not a signed/expiring URL): `stripResolvedImageUrls()` goes URL→path for outgoing `PATCH` calls, `resolveDraftImageUrls()` goes path→URL for `next/image` inside Preview (which throws on a raw path — it's not a valid `src`). Both share one field-walking function (`mapHotsiteImageFields()`) so a new module type with a new image field only needs adding to the field list once, not once per direction. This is the same "define the shape traversal once, parameterize the transform" pattern you'd reach for backend-side with a visitor — just applied to a plain JSON tree instead of a class hierarchy.

**Why Preview is a client-side view-swap, not a route.** The obvious way to build "preview what this looks like before publishing" is a separate route that renders the public page against draft data. That was rejected in favor of a third state (`'preview'`, alongside `'tabs'`/`'module-config'`) inside the same `HotsiteEditor` component, reached via a "Preview" button and exited via the shared dashboard topbar's back-arrow override mechanism — the exact same mechanism the module-config drill-down already used. Two reasons this won out: first, it means Preview shares the dashboard shell (sidebar, topbar) rather than needing its own chrome-less layout; second, and more subtly, it means Preview always renders the *in-memory* draft the admin is actively editing, with zero risk of it drifting from a separately-fetched copy — there's only one source of truth for "what does the draft currently look like," and it's the same React state the tabs are editing. The cost is that the M12 public render components (`HeroModule`, `ServiceListModule`, etc.) — originally written as Server Components with zero client JS cost on the public site — become client-hydrated code once imported into this `'use client'` editor tree. That cost is bounded by lazy-loading Preview via `next/dynamic(..., { ssr: false })`, so it's only paid by an admin who actually clicks "Preview," never by a visitor to the public hotsite itself.
