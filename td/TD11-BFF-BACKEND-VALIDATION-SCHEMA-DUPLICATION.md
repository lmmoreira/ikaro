## Problem

The BFF and backend each define their own Zod schema to validate the same request body. For `tenants/settings`, this only became visible as a literal SonarCloud finding after M13-S10 normalized the backend's `tenants.settings` JSONB from snake_case to camelCase: once both layers spoke the same casing, their schemas became textually near-identical (43%/59% duplication on the two files), tripping the 3% new-code duplication gate. Fixed for that PR with a `sonar.cpd.exclusions` entry — see `sonar-project.properties`.

The same pattern already exists for hotsite content (`HotsiteAdminController`'s branding/layout/seo schema vs. the backend's `update-hotsite-content.dto.ts`) — it just stays under the duplication threshold there because the schema is large enough that the percentage doesn't cross 3%, not because the architecture is different.

This is a real "two sources of truth" risk, independent of the SonarCloud metric: if a business rule changes on one side (e.g. backend extends `cancellationWindowHours`'s max from 720 to 1000) and the BFF's copy isn't updated in the same PR, the BFF will reject valid requests with a stale, wrong error before they ever reach the backend. Nothing currently catches this drift automatically.

TD23's discovery (2026-07-11) found this problem is worse than originally catalogued: `bookings.controller.ts` and `customers.controller.ts` each define their own `AddressSchema` (identical field shape, no country-awareness) plus their own `contactPhone`/`phone` E.164 regex and `contactEmail` checks — duplicating the backend's `Address`/`PhoneNumber`/`Email` VOs independently of the tenant-settings/hotsite instances above. That's four BFF-side Address copies total (the fourth being `BusinessInfoAddressSchema` in `tenant-settings.controller.ts`) plus at least three independent phone/email format copies. TD23 Story 10 assigns the backend's own error codes to these copies as an interim consistency fix (same code regardless of which layer rejects the request) — it does not deduplicate the schemas themselves, that remains this TD's scope.

Also found: `tenant-settings.controller.ts`'s `BusinessInfoSchema.phone`/`.email` fields, and `SocialLinksSchema.whatsapp`, have **no** format validation at all in the BFF today (bare `z.string().nullable()` / `z.string().nullable().optional()`), unlike every other phone/email field in the BFF — an invalid business phone/email/WhatsApp number currently round-trips to the backend before being rejected there. The backend already validates all three server-side: `apps/backend/src/contexts/platform/domain/value-objects/validators/business-info.validator.ts`'s `BusinessInfoValidator` calls `PhoneNumber.isValid`/`Email.isValid` (the same VOs) but throws `PlatformErrorCode.SETTINGS_BUSINESS_PHONE_INVALID`, `SETTINGS_BUSINESS_EMAIL_INVALID`, and `SETTINGS_SOCIAL_WHATSAPP_INVALID` respectively — **not** the generic `PhoneErrorCode.FORMAT_INVALID`/`EmailErrorCode.FORMAT_INVALID` codes used elsewhere in the BFF. When this TD's shared-schema package lands, the BFF-side checks for all three fields must reuse these exact `PlatformErrorCode` values, not the generic VO codes — picking the generic codes would recreate this TD's own "two sources of truth, mismatched codes" problem. This gap should be closed for real as part of the migration, not patched with another ad-hoc regex copy in the interim.

**A second, smaller duplication (found 2026-07-16):** it's not just the schema *content* that's duplicated — the Zod-to-NestJS *pipe mechanism* is too. `apps/backend/src/shared/http/zod-validation.pipe.ts` and `apps/bff/src/shared/http/zod-validation.pipe.ts` are near-identical: both take a `ZodType`, call `.safeParse()`, and map failures to a Problem Detail / `BadRequestException`. Unlike the schema-content duplication (business rules, out of place in an HTTP-transport package), this one is pure NestJS wiring with zero business content — the same category as `CanonicalParseUUIDPipe`/`CanonicalParseIntPipe`, which already live in `@ikaro/nestjs-http` (`packages/nestjs-http/src/canonical-parse-pipes.ts`) for exactly this reason ("Shared NestJS HTTP infrastructure (pipes, filters) for backend and bff — never consumed by apps/web"). This is scope this TD should also cover, but as a distinct, much smaller extraction — no new package needed, since `@ikaro/nestjs-http` already has the `dist`/`build`/Dockerfile wiring in place.

## Why not fixed now

Properly deduplicating means extracting the shared Zod schema into a new package both `apps/backend` and `apps/bff` import — not `@ikaro/types` (that's the BFF→frontend contract; backend importing it would violate the anti-pattern already in `docs/ANTI_PATTERNS.md` about coupling the backend domain to the frontend-facing shape). A new package with real runtime code (Zod schemas execute, they're not type-only) needs the full treatment already documented for `packages/*` runtime packages: a `build` script compiling to `dist/`, a root `postinstall` so `dist/` is fresh locally, and **both apps' Dockerfiles updated** to build it before `pnpm deploy --prod` (a previous package skipped this and broke production with `ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING`). That's new package scaffolding + build-chain wiring + two Dockerfile changes + verification — disproportionate to bundle into the PR that happened to surface the metric, and it should cover hotsite too for consistency rather than fixing one instance ad hoc.

The `ZodValidationPipe` mechanism duplication (see Problem) does **not** carry the same cost — `@ikaro/nestjs-http` already exists, already has `dist`/`build`/`postinstall`/Dockerfile wiring for both apps, and already hosts the same category of code (`CanonicalParseUUIDPipe`/`CanonicalParseIntPipe`). Consolidating the pipe there is a same-shape, low-risk extraction independent of the schema-content package below, and could be picked up on its own if this TD's full scope stays deferred.

## Proposed fix (not yet scoped as a story)

**Track A — mechanism (`ZodValidationPipe`), low effort, no new package:**

1. Move a generic `ZodValidationPipe<T>` into `@ikaro/nestjs-http`, alongside `CanonicalParseUUIDPipe`/`CanonicalParseIntPipe`; delete `apps/backend/src/shared/http/zod-validation.pipe.ts` and `apps/bff/src/shared/http/zod-validation.pipe.ts` and update every call site to import from `@ikaro/nestjs-http`.

**Track B — schema content, needs a new package:**

2. Decide where shared backend↔BFF validation schemas should live (new package, e.g. `packages/contracts` or `packages/validation` — name TBD) and confirm it's distinct in purpose from `@ikaro/types` (per `docs/ANTI_PATTERNS.md`'s "Backend importing `@ikaro/types` for response/DTO shapes" row — business validation content is not one of that rule's two carved-out exceptions).
3. Add the package with compiled `dist/` output, `build` script, `postinstall`, and wire both `apps/backend/Dockerfile` and `apps/bff/Dockerfile` to build it before deploy.
4. Migrate `update-tenant-settings.dto.ts` + `tenant-settings.controller.ts`'s schemas to import from the shared package; remove the `sonar.cpd.exclusions` entry once duplication is gone for real (and note: fix or remove that entry regardless, since its path is already stale against the current file location — see Status).
5. Migrate `update-hotsite-content.dto.ts` + `hotsite-admin.controller.ts`'s schemas the same way for consistency.
6. Decide whether the shared schema's defaults/refinements differ at all between the two layers (e.g. does the BFF need `.strict()` if the backend doesn't, or vice versa) before assuming a single export works for both call sites unmodified.
7. Migrate `bookings.controller.ts` and `customers.controller.ts`'s `AddressSchema` copies (and their `contactPhone`/`phone`/`contactEmail` checks) the same way — 4 Address copies + 3+ phone/email copies total once fully centralized.
8. Include `businessInfo.phone`, `businessInfo.email`, and `socialLinks.whatsapp` format validation in the migrated shared schema (all three currently unvalidated at the BFF layer — see Problem section) rather than leaving that gap in place. Reuse the backend's exact codes for each — `PlatformErrorCode.SETTINGS_BUSINESS_PHONE_INVALID`, `SETTINGS_BUSINESS_EMAIL_INVALID`, `SETTINGS_SOCIAL_WHATSAPP_INVALID` (`business-info.validator.ts`) — not the generic `PhoneErrorCode`/`EmailErrorCode` VO codes.
