# Ikaro — Anti-Patterns Reference (BLOCK MERGE)

> Canonical list. Loaded automatically by `/pre-pr`. See `CLAUDE.md §8` for the 9 highest-severity ones kept inline.

| Pattern | Problem | Fix |
|---|---|---|
| `WHERE id = ?` without `tenant_id` | Cross-tenant data leak | Add `AND tenant_id = ?` |
| Event missing `tenantId` in envelope | Can't isolate per tenant | Include in every event |
| Hardcoded `48`, `180`, `7` for business rules | Breaks per-tenant config | Read from `tenants.settings` |
| `@ts-ignore`, `any`, `eslint-disable` | Defeats static analysis | Fix the type/lint error |
| `.skip()` / `.only()` in tests | Hides failures in CI | Remove before commit |
| Synchronous call from Loyalty → Booking | Tight coupling | Subscribe to `BookingCompleted` event |
| `new XRepository()` inside a service | Untestable | Inject via DI |
| Same template body for all tenants | Breaks branding | Templates are per-tenant aggregates |
| Photo stored at `bookings/<id>/` without tenant prefix | No isolation | Path: `tenants/<tid>/bookings/<bid>/<file>` |
| Logging without `tenant_id` | Can't slice per-tenant | Add to structured log context |
| Running migrations at app startup | Unsafe for rolling deploys | Run as separate CI job before deploy |
| English copy in email templates | Wrong locale | All customer-facing text in pt-BR |
| Money as plain `number` | Loses currency | Use `Money { amount: Decimal, currency: 'BRL' }` |
| Import from `src/contexts/<B>/` inside Context A | Breaks context isolation | Only import from `src/shared/` or own context |
| SQL JOIN into another context's schema inside a repository | Hardest coupling — defeats schema independence, blocks port swapping | Repository queries its own schema only; cross-context data via events, BFF, or port+adapter |
| Cross-schema DB FK between contexts | Tight schema coupling | Store UUID only; no FK constraint across schemas |
| Event consumer querying another context to fill missing data | Defeats self-contained events | Add the needed data to the event payload |
| Placing a domain entity or use case in `src/shared/` | Blurs context ownership | Only ports, base classes, and multi-context VOs in shared |
| Exporting repository tokens from a `*.module.ts` | Makes repo injectable cross-module — BC isolation violation | Never export repository tokens; use BFF orchestration, events, or shared read-only port |
| Calling `save()` in a use case without `ITransactionManager.run()` | TypeORM's `save()` is a merge (internal SELECT + UPDATE/INSERT) — not atomic without a transaction. Single-aggregate writes have this problem too. | Wrap every `save()` in `txManager.run(async () => { await repo.save(entity); })`. Reads and business logic stay **outside** the transaction to keep it short. |
| Writing to two or more aggregates without `ITransactionManager.run()` | Partial DB failure leaves inconsistent state across aggregates | Wrap all saves together in one `txManager.run(async () => { ... })` call |
| Using `jest.fn()` to stub `IEventBus` or `ITransactionManager` | Misses state assertions; mock expectations are brittle | Use `InMemoryEventBus` / `InMemoryTransactionManager` from `src/test/infrastructure/` |
| Multiple `if (err instanceof X)` chains inside a controller method | Noisy, inflates cognitive complexity | Extract into a `mapXxxError(err: unknown): never` helper in `infrastructure/http/` |
| Defining a role guard (`XxxRoleGuard`) inside a context's `infrastructure/guards/` | Role guards enforce the shared `X-Actor-Role` header contract and are used across multiple contexts — they belong in `src/shared/guards/`, not inside a context | Move to `src/shared/guards/` and update all imports; context-level `infrastructure/guards/` is reserved for guards with domain-specific logic (e.g. `PlatformAdminGuard` with `ConfigService`) |
| Importing a guard from another context's path (e.g. `../../../booking/infrastructure/guards/`) | Hard violation of context isolation rule — context A must not import from context B's path | Role guards must be in `src/shared/guards/`; any remaining context-specific guard stays local |
| Barrel `index.ts` in `ports/` or `shared/domain/` directories | Hides symbol origins; circular dep risk | Import directly from the specific file; `no-restricted-imports` ESLint rule enforces this |
| Single `DATABASE_URL` connection string for TypeORM | Passwords with special chars (`@`, `:`, `/`) break silently | Use five explicit vars: `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` |
| `TypeOrmModule.forRoot({ … process.env['X'] … })` | Env vars are `undefined` at import time (before dotenv) | Use `TypeOrmModule.forRootAsync({ useFactory: () => ({ … }) })` |
| `{{$env varName}}` in `.http` REST Client files | Reads OS env — resolves to empty string | Use `{{varName}}` for REST Client env vars; `{{$dotenv VAR}}` only for `.env` secrets |
| Duplicating a validation function (`isValidEmail`, etc.) across files | Two copies diverge silently | Extract into a shared value object in `src/shared/value-objects/` |
| Storing email, phone, address, money, colour as a plain `string` / `number` | Invalid values reach the domain silently | Wrap in a value object that validates on construction |
| Duplicating a utility function (deep merge, formatting, etc.) | Two copies diverge | Extract into `src/shared/utils/` |
| Aggregate field typed as `string` when a shared VO exists | Type system lies — invalid values stored in props | Type props with the VO; getters return the VO |
| `makeEntity()` helper or plain object literal used in a test | Couples test to TypeORM entity constructor; bypasses builder pattern | Create an `XxxEntityBuilder` in `src/test/builders/<context>/` |
| Seed file calling `CREATE TABLE`, `CREATE SCHEMA`, or `DROP TABLE` | Drift from migrations | Seeds are data-only; schema owned 100% by TypeORM migrations |
| `XxxEntityBuilder` with a hardcoded default `id` | Second `save()` silently upserts over first — isolation assertions fail | Default `id` to `uuidv7()` in every `XxxEntityBuilder` constructor |
| Controller directly injecting a repository token | Bypasses use-case layer — domain errors never thrown, HTTP mapping skipped | Controllers inject use cases only |
| `jest.fn()` mock for a use case in a controller unit test | Hides real behaviour; tests only delegation | Wire the real use case with `InMemoryXxxRepository` |
| `makeController()` / `makeUseCase()` / `make()` factory function used to wire spec dependencies | Factory re-creates everything per `it()` call; repo/double refs returned from the factory are not accessible in sibling `it()` blocks without re-calling it — inconsistent with the rest of the codebase and detected by code review | Always use `let` declarations at `describe` scope + `beforeEach` to initialize; `it()` bodies seed data through those `let` refs (e.g. `await repo.save(...)`) |
| Exporting constants or helpers from `main.ts` | `import { X } from '../main'` triggers bootstrap / `process.exit(1)` in tests | Move shared constants to a dedicated module file; `main.ts` may re-export them |
| Inline `schema.safeParse(body)` inside a controller method | Inconsistent with `ZodValidationPipe` + DTO pattern; loses typed `@Body()` | Define schema + `z.infer<>` type in `application/dtos/`; apply `@UsePipes(new ZodValidationPipe(schema))` |
| Controller or route file contains branching, policy, or response shaping | Composition layer is doing application work | Move the branch or mapping into the owning slice's use case, helper, or mapper |
| `dangerouslySetInnerHTML` is used inline in a page or reusable component | Raw HTML injection is easy to forget to sanitize | Centralize it behind a controlled wrapper/helper with an explicit sanitization path |
| `z.string().uuid()` / `z.string().url()` | Deprecated in Zod v4 (SonarCloud S1874); `z.uuid()` rejects non-RFC-4122 test UUIDs | Use `z.uuid()` and `z.url()` directly; use RFC 4122-compliant UUIDs: `'10000000-0000-4000-8000-000000000001'` |
| Declaring a dynamic route (`@Get(':id')`) before a static route | NestJS resolves in declaration order — dynamic matches first | Always declare static/prefix routes first, then parameterized ones |
| Use case returns `null` / `undefined` instead of throwing a domain error | Controller must inspect the return value and decide HTTP status — business logic in the wrong layer | Use cases always throw domain errors for every non-happy-path; controller is one line: `return this.useCase.execute(dto).catch(mapXxxError)` |
| Throwing `HttpException` directly from a use case | Couples the application layer to HTTP — use cases must be framework-agnostic | Throw domain errors only; `mapXxxError` converts them to `HttpException` |
| Non-UUID string (e.g., `'non-existent-id'`) as path/query param for a PostgreSQL UUID column | PostgreSQL throws `QueryFailedError: invalid input syntax for type uuid` → 500 instead of expected 404/400 | Add `ParseUUIDPipe` to every `@Param`/`@Query` that maps to a UUID column; use valid-UUID-format IDs in integration tests |
| Integration test `it()` with only supertest `.expect(status)` and no Jest `expect()` call | SonarCloud S6957 BLOCKER — supertest's `.expect()` is invisible to Jest's assertion counter | Every `it()` must have at least one `expect()` call |
| `.catch(() => null)` on BFF backend HTTP calls | Swallows 5xx errors and timeouts — backend outage silently misdirects users | Only catch the expected failure status; rethrow everything else |
| `new Error('msg')` to mock `BackendHttpService` errors in BFF tests | Plain `Error` is not caught by `instanceof HttpException` checks | Mock errors as `new HttpException('Not Found', 404)` |
| Use case `execute()` return type named `*Info`, `*Dto`, raw `T[]`, or any ad-hoc name | Callers can't predict the type name | Name the result `{UseCaseClassName}Result`; define and export it in the same `.use-case.ts` file |
| Request DTO named `{Action}RequestDto`, `{Action}InputDto`, or any suffix other than `Dto` | Inconsistent naming | Use `{Action}Dto` only; Zod schema is `{Action}Schema` |
| Split DTO pattern: `{Action}RequestDto` (body) + `{Action}Dto` (body + path param) merged via `{ pathParam, ...dto }` | Two types for one use case input — unnecessary complexity | Pass path param and body as separate arguments: `execute(pathParam, dto)` |
| New domain error class added to `xxx-domain.error.ts` but not added to `mapXxxError` | Error falls through to the generic `XxxDomainError` catch → wrong HTTP status (e.g. 400 instead of 422) with no test failure | Every new `XxxSpecificError` must have its own `if (err instanceof XxxSpecificError)` branch in `mapXxxError` before the generic base-class catch |
| BFF response interface declared inline in a controller file (e.g. `export interface CancelBookingResponse { … }` inside `bookings.controller.ts`) | Inconsistent with the project convention; other response shapes in the same feature live in `xxxs.types.ts` | All API response interfaces for a BFF feature go in `apps/bff/src/xxxs/xxxs.types.ts`; import from there |
| BFF module/controller named after a single aggregate or resource instead of its bounded context (e.g. `tenants/` for the Platform context, which owns `Tenant` + `HotsiteConfig`) | Misleading name forces a rename later when the mismatch is noticed (e.g. M12-S05 `tenants/` → `platform/`) | Name BFF modules after the bounded context (CLAUDE.md §3); split controllers by audience — `<context>.controller.ts` (admin) vs `<context>.public.controller.ts` (`@Public()` hotsite) |
| `@UseGuards(InternalApiGuard)` on individual controllers instead of `APP_GUARD` in `AppModule` | New controllers silently skip the guard — no compile error, no test failure | Register `{ provide: APP_GUARD, useClass: InternalApiGuard }` in `AppModule.providers`; every current and future controller is automatically protected |
| Adding `X-Internal-Key` to `buildBackendHeaders()` via `process.env` reads | `buildBackendHeaders` is a pure request-scoped function — reading `process.env` makes it untestable and bypasses DI | Add the key at `BackendHttpService` level: merge into `headers()` return value and add to each `*ForPublic` inline header object via `this.config.getOrThrow('INTERNAL_API_KEY')` |
| NestJS `StorageModule` (or any shared module) registering an adapter with `useExisting`: `providers: [GcsSignedUrlAdapter, { provide: STORAGE_SERVICE, useExisting: GcsSignedUrlAdapter }]` | `useExisting` creates an alias but `GcsSignedUrlAdapter` is still registered as its own standalone class provider and is instantiated unconditionally — overriding `STORAGE_SERVICE` in tests only removes the alias; the standalone class (and its `onApplicationBootstrap` network calls) still runs | Use `useClass`: `providers: [{ provide: STORAGE_SERVICE, useClass: GcsSignedUrlAdapter }]` — NestJS only instantiates the adapter when the token is requested; overriding the token in tests fully prevents instantiation |
| Integration app helper (e.g. `createBookingIntegrationApp`) does not default-override adapters that call external services in `onApplicationBootstrap` | Any test that uses the helper without explicitly overriding the token will hit the real adapter — causes `ECONNREFUSED` locally and potentially in CI if the external service is absent | Every integration app helper must default-override such adapters with their in-memory stub (e.g. `.overrideProvider(STORAGE_SERVICE).useValue(new InMemoryStorageService())`); callers that need a specific stub can override again via `overrideProviders` — the last override wins |
| Backend importing `@ikaro/types` for response/DTO shapes | `@ikaro/types`'s response interfaces are the BFF→Frontend contract; backend importing them couples the domain layer to the frontend API shape — a frontend rename forces a backend change | Backend defines its own DTOs; BFF maps backend responses to `@ikaro/types` shapes before returning to callers. **Exception:** `packages/types/src/error-codes.ts`'s per-origin code catalogs (`BookingErrorCode`, etc.) are a deliberate cross-cutting identifier catalog, not a response shape — TD23 imports these directly into backend domain error classes as the single source of truth shared with BFF/web, preventing string-literal drift between layers. **Second exception:** `packages/types/src/errors.dto.ts`'s `ProblemDetail`/`ValidationViolation`/`ValidationProblemDetail` are the RFC 9457 error envelope (CLAUDE.md §1: "Errors: RFC 9457 Problem Details on all non-2xx") — a standardized wire-format contract every layer's HTTP-adapter code must speak identically, not a feature-specific response shape driven by frontend display needs. Every backend consumer lives in `shared/http/` or `infrastructure/guards/` — the HTTP-adapter layer, whose job is to speak the external protocol. TD23 Story 8 found and fixed a byte-identical hand-copied duplicate of this exact type (`apps/backend/src/shared/http/problem-detail.ts`), which is the actual anti-pattern this row exists to prevent — a single canonical envelope avoids re-introducing it |
| A Zod schema check duplicates a VO's own validation (e.g. `.refine(Email.isValid, ...)` in a DTO or BFF schema, mirroring the `Email` VO's own check) | The same business rule gets two independently-assigned codes depending on which layer catches the violation first — the frontend shows an inconsistent message for the identical violation depending on which layer rejected it | Reuse the VO's own error code (e.g. `EmailErrorCode.FORMAT_INVALID`) in the Zod check instead of minting a new one; only checks with no VO behind them share a small closed `GenericErrorCode` set — see `docs/ENGINEERING_RULES.md` § Single source of truth for a validation rule's code |
| `as React.CSSProperties` on a function returning CSS custom properties | SonarCloud flags it as an unnecessary assertion; also loses type precision on the custom property keys | Return `React.CSSProperties & Record<\`--ba-${string}\`, string>` — the intersection type satisfies both the CSS property contract and allows custom `--ba-*` keys without a cast |
| `<Image fill>` in Next.js without a `sizes` prop | Next.js cannot generate responsive srcsets — browser always downloads the full-resolution image regardless of viewport; emits a runtime console warning | Add `sizes` matching the element's layout role: `"100vw"` for full-width sections, `"(min-width: 640px) 50vw, 100vw"` for half-column images, `"(min-width: 768px) 33vw, 100vw"` for thirds |
| `import '@testing-library/jest-dom'` (bare) in `vitest.setup.ts` | Vitest-specific matchers (`toBeInTheDocument`, `toHaveAttribute`, etc.) may be untyped — `pnpm type-check` is silent but IDEs and strict checks report missing type declarations | Always import `@testing-library/jest-dom/vitest` — the `/vitest` entrypoint registers types for Vitest's `expect()`; the bare form is Jest-only |
| `<dialog className="flex ...">` (any Tailwind display utility on a `<dialog>`) | Tailwind's utility (`display: flex`) in the author stylesheet overrides the UA's `dialog { display: none }` — the closed dialog is always visible as a transparent fullscreen overlay; its `position: fixed; inset: 0` backdrop element silently intercepts all page clicks, making the entire page non-interactive | Use `open:flex` — Tailwind's `open:` variant applies only when the element has the `open` attribute (set by `showModal()`); never put `flex`, `block`, `grid`, or any other display utility directly on a `<dialog>` |
| `export const revalidate = SOME_IMPORTED_CONSTANT` in a page or layout | Next.js extracts segment config (`revalidate`, `dynamic`, `runtime`, etc.) via static AST analysis — the module is never executed, so imported variable references are not resolved; `next build` fails with "Invalid segment configuration export detected" | Use a numeric literal: `export const revalidate = 300;` with a comment pointing to the shared constant; the constant itself can still be used in `fetch()` calls which run at runtime |
| New `app/[slug]/*` page with no `backgroundColor: 'var(--ba-background)'` on its root wrapper | `HotsiteLayout` sets `--ba-*` CSS variables on `#hotsite-root` but never applies them as actual background/color styles — `globals.css` keeps `body { background: #ffffff }`, which shows through any transparent wrapper; dark-themed hotsites silently become white pages with invisible text | Wrap every new booking/sub-page root in `<div style={{ backgroundColor: 'var(--ba-background)', color: 'var(--ba-text)' }}>` |
| New `packages/*` workspace package with real runtime code ships `"main"`/`"types"` pointing at `src/*.ts` | Only safe while every export is used exclusively in type positions across all consumers (erased at compile time, never `require()`'d). The moment any consumer imports a real value (a const, a class, an enum-like object) from it, production's `node dist/main.js` crashes with `ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING`: Node's native TS type-stripping explicitly refuses `.ts` files resolved through `node_modules`. `docker build`, `tsc --noEmit`, and every test pass — only `docker run` reveals it. (`packages/types` made exactly this transition in TD23 Story 2/3, when its previously type-only `BookingErrorCode`/`AddressErrorCode`/etc. catalogs became real values imported by the backend — don't assume "type-only today" means "type-only forever") | Ship compiled JS: add a `"build": "tsc"` script, point `"main"`/`"types"` at `dist/index.js`/`dist/index.d.ts`, add a root `postinstall` so `dist/` is always fresh for local/CI, and have every Dockerfile that depends on it build it explicitly before `pnpm deploy --prod` |
| A package directly imported in production (non-spec) `src/` code is listed only in `devDependencies`, or not listed at all (only available transitively) | `pnpm deploy --prod` strips `devDependencies` entirely and never hoists transitive-only packages to a consumer that doesn't directly depend on them — dev and test never notice since they see the full dependency graph regardless of classification; the app silently fails to boot in production with `Cannot find module 'x'` | Before adding any new production import, confirm it's in `dependencies`. To audit an app wholesale: `grep -rhoE "from '[^.][^']*'" src --include="*.ts" \| grep -v spec` then cross-check each package name against the `dependencies` block — caught 3 separate real cases (`express`, `jsonwebtoken`, `ms`) in one sweep |
| Creating a new cross-context Port+Adapter when one already exists for the same context pair (e.g. `ITenantLocalizationPort` alongside `IBookingPlatformPort`, both booking→platform) | Duplicate adapters drift apart, double the DI wiring, and the new one often ships without its own `.spec.ts` — went unnoticed through two merged PRs (TD02-S02, TD02-S03) until TD02-S04 | Grep `infrastructure/cross-context/` for an existing adapter between the same two contexts first; add a method to it instead of creating a parallel port |
| Shared VO's `create()` throws a plain `Error` for a rule the DTO's Zod schema can't fully replicate (e.g. `Address` validates against a country-specific `CountrySpec.postalRegex`, but the DTO can only check `.min(1).max(20)`) | Every `mapXxxError` ends with `if (err instanceof Error) throw err;` — a plain `Error` falls through that unchanged and becomes an unhandled 500 instead of a 400 | Give the VO a typed error class (e.g. `AddressValidationError`, with `Object.setPrototypeOf`); add an `instanceof` branch to every calling context's error mapper that invokes the VO's `create()` |
| Shared infrastructure (a repository, or any class called from more than one invocation context) reads `RequestContext` directly, e.g. `this.requestContext.settings` | `RequestContext`'s `AsyncLocalStorage` store is populated only by `RequestInterceptor`, which runs in the HTTP request pipeline only — the same repositories are also called from cron jobs (`*.job.ts`) and event handlers (via cross-context adapters), neither of which run inside that interceptor; reading any `RequestContext` getter outside its `.run()` scope throws `Cannot read properties of undefined` | Inject a `tenantId`-parameterized port instead (e.g. `ITenantSettingsPort.getSettings(tenantId)`) and pass the method's own `tenantId` argument; reserve direct `RequestContext` reads for use cases, which only ever run inside one invocation context (the HTTP request) |
| `InMemoryXxxPort` silently ignores a method parameter that scopes results (e.g. `locale` in `getEmailTableHeaders`) | Tests pass even when production code calls the port with the wrong value — e.g. `getEmailTableHeaders('adminDailySchedule', 'en')` returns the same headers as `'pt-BR'`, so a bug that passes the wrong locale is never caught in unit tests | When a port method takes a scoping parameter, the in-memory double must use it as part of the storage key — e.g. `setTableHeaders(tableKey, locale, headers)` stores under `tableKey + locale`; `getEmailTableHeaders(tableKey, locale)` looks up by both. The double should throw for missing key combinations to mirror the real adapter's behaviour |
| `<Context.Provider value={{ a, b, c }}>` without `useMemo` in a `'use client'` provider component | Creates a new object on every render — all consumers of the context re-render on every parent render, even when `a`, `b`, `c` haven't changed; SonarCloud flags this as MAJOR | Wrap in `useMemo`: `const value = useMemo(() => ({ a, b, c }), [a, b, c]); return <Context.Provider value={value}>` |
| Declaring a new request/response interface in `apps/web/features/**/api/**` or `apps/web/shared/lib/api/**` without checking `@ikaro/types` first | Either duplicates a correct shared type (drifts silently — found in 5 BFF modules, `td/TD09-WEB-TYPES-DRIFT-VS-IKARO-TYPES.md`) or, if `@ikaro/types` is itself stale, blindly importing it ships a wrong contract | Grep `@ikaro/types` for the same/similar name; if shapes differ, verify against the live BFF schema which side is correct before picking one |
| Inline `to*` response-mapping functions accumulating in a BFF `*.controller.ts` | Controller grows untestable in isolation — mapping logic can only be exercised through full HTTP-mock setup | Once a second mapper function appears, extract both into a sibling `<module>.mapper.ts` (plain functions, not a class) with its own direct-call spec |
| A second, independent `fetch(...)` call building the same BFF route URL outside the canonical `lib/api/<name>.ts` fetcher (e.g. `lib/i18n/resolve-locale.ts` building its own `/platform/manifest/:slug` URL instead of reusing `lib/api/platform.ts`) | Renaming/moving the route updates the canonical fetcher but silently misses the duplicate; the stale URL 404s and falls back to a default, masked because the duplicate's own unit test asserts `fetch` was called with the same (now-stale) hardcoded URL — internally consistent, never checked against the real route. Only caught by Playwright hitting the real BFF (M13-S05) | Extract a shared low-level `fetch<X>Response(...)`/raw-`Response` helper in the canonical `lib/api/<name>.ts`; each caller layers its own error handling on top (e.g. `notFound()`/throw for page use vs. a soft fallback for i18n config use) instead of building the URL independently |
| A cross-context port/use-case pre-formats structured data into a single display string before any consumer is known (e.g. `booking.lines.map(l => l.name).join(', ')` inside `LoyaltyBookingAdapter`, or worse, picking only `lines[0]`) | Bakes a presentation decision into the backend layer too early; picking only the first item silently drops information at the data-retrieval layer, and even joining early removes the BFF's ability to reorder/filter/re-render differently later (M13-S08 — a redemption→service-name lookup for `bookingReference`) | Return structured data (e.g. reuse an existing `ServiceSummary[]`-style shape) all the way through the port → use case → backend HTTP response; do the actual join/format in the BFF mapper (`<module>.mapper.ts`), which already owns customer-facing presentation shaping |
| `PATCH /tenants/settings` accepting a field that belongs to a different resource (e.g. `name`, when `name`/`slug` are separate `varchar` columns on `tenants`, not part of the `settings` JSONB) | Conflates two independent resources behind one contract; a `.strict()` validation schema on the settings shape can't say anything about the identity fields riding along, and a consumer reading the route has no signal that it also mutates unrelated columns | Split into resource-accurate endpoints — one per actual resource (e.g. `PATCH /tenants` for identity, `PATCH /tenants/settings` for the JSONB blob) — mirror the split 1:1 at the BFF layer too rather than re-merging it into one orchestrated endpoint |
| Duplicate read endpoints/use cases that return projections of the same aggregate/config (e.g. `GET /tenants/formatting` or `GET /tenants/booking-config` alongside `GET /tenants/settings`) | Creates parallel contracts, duplicate tests, duplicate DI wiring, and blocks simple caching because each projection re-reads the same source of truth. The smaller response shape feels convenient for one caller but becomes stale the moment another caller needs a neighbouring field. | Keep one canonical read endpoint/use case for the source of truth (e.g. `GET /tenants/settings` via `GetTenantByIdUseCase`). Derive caller-specific values in the BFF mapper or web helper. A separate read endpoint is justified only when the canonical payload would expose data the caller must not see, or when the endpoint composes genuinely different sources. |
| Exported `*QueryService` wrappers used only by cross-context adapters (e.g. `TenantQueryService`, `CustomerQueryService`) | Creates a second application API beside use cases, often as repository pass-through plumbing. Callers then depend on aggregate internals or ask for super-narrow projections like "manager emails", which multiplies one-off readers instead of establishing a reusable read contract. | Cross-context adapters depend on exported read use cases from the source context. Use `Get<Entity>ByIdUseCase` for single lookup; for list/search reads prefer broader `Get<Entities>UseCase` with a filter DTO (`ids`, `status`, `roles`, `search`, pagination). Avoid super-narrow use cases such as `GetManagerEmailsUseCase`; return a useful DTO and let the caller map the field it needs. If the right breadth is unclear, discuss the read contract before adding a new use case. |
| Creating a second use case only to reshape the response for one caller (e.g. `GetCustomerProfileUseCase` duplicating `GetCustomerByIdUseCase`, or a smaller booking summary use case beside `GetBookingByIdUseCase`) | Duplicates application logic just to satisfy a caller-specific contract and splits the source of truth for the same aggregate read. | Keep one canonical read use case and map the caller-specific shape in the controller, adapter, BFF mapper, or client helper. Extract a dedicated DTO/mapper only if the shaped output is reused in more than one place or needs to be shared as a type. |

| A class implements multiple ports (e.g. `IEventBus` + `ITriggerBus` + `IPushableEventBus`) solely because several DI tokens all happen to alias the one functional token this class will occupy | The interface bundle came from token wiring, not from the class's actual responsibility — it forces unrelated methods onto a class with no behavioral need for them (an ISP violation). Found in TD24-S01: `OutboxEventBus` (later renamed `OutboxPublisher`) carried `subscribe()`, `registerTrigger()`, `publishTrigger()`, `dispatchPushMessage()` as pure pass-throughs purely because `TRIGGER_BUS`/`PUSHABLE_EVENT_BUS` aliased `EVENT_BUS`, which this class was going to occupy — none of those four methods had anything to do with the outbox's actual job (durably recording an envelope in `shared.outbox`) | Before implementing every interface a class's target token seems to require, trace each interface's methods back to whether they're actually invoked through the *same* token being rebound. If a method is only reachable through a separate, never-rebound token, point that token straight at the concrete adapter instead of down through the new class (see TD24 D12) |
| A class's own class-token DI registration is `{ provide: SomeClass, useExisting: FUNCTIONAL_TOKEN }`, where `FUNCTIONAL_TOKEN` is expected to be rebound to a different implementation later | The class token silently starts resolving to whatever `FUNCTIONAL_TOKEN` gets rebound to instead of staying pinned to the real class — any consumer injecting by the class token (expecting the original concrete implementation) gets the new implementation instead, with no compiler error. Near-miss in TD24-S01: `GcpPubSubEventBusAdapter`'s class token aliased `EVENT_BUS`; a later story rebinding `EVENT_BUS` to a different class would have silently broken `OutboxRelayService`'s injection of the real Pub/Sub adapter. Distinct from the `useExisting`-adapter-token anti-pattern above (§8's "double instantiation" case) — this is about alias *direction*, not duplication | Make the class its own bare provider (the anchor, e.g. plain `SomeClass` or `{ provide: SomeClass, useClass: SomeClass }` in `providers`); have the functional token(s) `useExisting` alias *to* the class token, never the reverse. A future rebind of the functional token then only ever touches that one line |
| An existing DI token gets silently rebound to an implementation with a materially different reliability/latency contract than callers were written against, so that no call site needs to change | Passes TypeScript's structural check (same interface, same method signatures) but violates the *behavioral* half of Liskov substitution — a caller reading `@Inject(EVENT_BUS) eventBus: IEventBus` has no signal that `publish()` might now take up to 5 minutes and carry retry/dedup semantics instead of hitting the transport immediately. TD24's original design planned exactly this: rebind `EVENT_BUS` from the direct Pub/Sub adapter to `OutboxPublisher` so all ~21 publish call sites wouldn't need to change. Reversed in D14 once it became clear D6 (already-planned repository auto-flush) had already shrunk the real call-site count to ~6, at which point an honestly-named explicit dependency cost almost nothing extra | When swapping in an implementation with a fundamentally different contract (durability guarantee, latency, failure mode — not just a different backend for the same contract), give it its own explicitly-named port instead of rebinding a token callers already depend on for the old contract. Reserve token rebinds for same-contract substitutions (e.g. swapping a real adapter for a test double) |
| A new call site that needs durable/outbox-backed publish (a new cron job, a new non-aggregate publish site, a new use case) injects `EVENT_BUS` out of habit instead of `OUTBOX_PUBLISHER` | Per D14, `EVENT_BUS` is permanently the direct-to-Pub/Sub path with no dedup/durability, so injecting it where durability was intended silently loses the outbox's guarantees. A type-level narrowing (`IEventBus.publish()` accepting only `DomainEvent`, rejecting `Command`) was tried and reverted the same day — it closed the cron-job case at the compiler, but forced `OutboxRelayService`'s relay-payload cast to mislabel a relayed `Command` as `DomainEvent`, which is dishonest for the one class that legitimately needs to handle both generically | Structural, not type-level: once every one of the 21 original publish sites is migrated off `EVENT_BUS` (S02 aggregate repos + S03 jobs), add an ESLint `no-restricted-imports` override forbidding `contexts/**/application/**` and `contexts/**/infrastructure/repositories/**` from importing `shared/ports/event-bus.port` at all — see td/TD24-OUTBOX-INBOX-PATTERN.md D14 and S03's acceptance criteria for the exact allow-list (16 handler files, `OutboxRelayService`, `shared/infrastructure/event-bus/**`, test doubles) |
| A new `@Global()` NestJS module is added so other modules don't need to import it, but its providers aren't listed in `exports:` | `@Global()` only controls whether *other modules* need an `imports:` entry to reach this module's exports — it does not itself make any provider injectable. A provider left out of `exports:` stays private to the module even though it's marked global, and any consumer gets "Nest can't resolve dependencies" with no hint that `@Global()` was the misleading signal. Found in TD24-S02: `OutboxModule` was marked `@Global()` when `OUTBOX_PUBLISHER` gained real consumers, but the `exports: [OUTBOX_PUBLISHER]` line was never added — every integration-app test helper and controller spec that needed `OUTBOX_PUBLISHER`/`OUTBOX_REPOSITORY` failed to resolve it, and the error pointed at the *consuming* module, not the missing `exports:` line, which delayed finding the root cause | Whenever a module is marked (or re-marked) `@Global()` because a token gained new consumers, verify the token is in that module's own `exports:` array in the same change — `@Global()` and `exports:` are two independent switches, both required |
| A test double models deferred/transactional behavior (e.g. dispatch only after a commit) but records its own observability state (a `.published`/`.calls` array) eagerly at the call site, before the deferred point | The double's bookkeeping no longer matches what it's simulating: if the surrounding transaction later rolls back (a later step in the same `txManager.run()` throws), the real outbox row is undone but the double's array still shows the event as recorded — a false-positive signal to any test asserting on it. Found in TD24-S02: `RoutingInMemoryEventBus.publish()` pushed to `.published` synchronously at call time, even on the path deferred via `scheduleAfterCommit()` for ambient-transaction callers; introduced by this same story's own change (draining events *inside* the transaction) and only reasoned through, not caught by a failing test, since no existing spec happened to roll back a multi-save transaction after the first event was recorded | Record the double's own observable state at the same point the thing it's simulating actually becomes durable/dispatched (e.g. inside the deferred callback), not at the initial call — so a rollback and a commit produce genuinely different, correct observable outcomes |
| An integration test asserts an aggregate call count (`toHaveBeenCalledTimes(N)`) on a mock whose real counterpart is invoked by a full-table sweep with no per-test scoping | The sweep (e.g. the outbox relay's scheduled sweep, `SKIP LOCKED` over the whole shared table) has no awareness of which test created which row — a leftover unpublished row from another concurrently-running test file against the same shared Testcontainers instance legitimately adds extra calls, making the aggregate count flaky or simply wrong, independent of any real bug.
